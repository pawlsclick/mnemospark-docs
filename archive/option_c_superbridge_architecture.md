# Option C – Base USDC → Superbridge → Uniswap → Crust\n+

**Status:** Exploratory architecture note (for reference) \n+**Context:** Replace AWS S3 / Orgs for MVP with IPFS + Crust storage, while keeping **USDC on Base** as the only user-facing currency. \n+**Goal:** Document Option C, where USDC on Base is converted to CRU (for Crust) using on-chain tooling, with the **bridge leg handled by Superbridge**.\n+\n+---\n+\n+## 1. High-level idea\n+\n+- Users and agents **only ever pay in USDC on Base**, via x402 from their Base wallet.\n+- Behind the scenes, mnemospark (or a backend/relayer) converts some of that USDC into **CRU on Crust** to pay for storage orders (pinning CIDs) on Crust.\n+- **Bridging step (Base → Ethereum USDC)** is handled by **Superbridge** ([docs.superbridge.app](https://docs.superbridge.app/)), using **Circle CCTP native USDC** ([Native USDC](https://docs.superbridge.app/native-usdc)).\n+- **Swap step (Ethereum USDC → CRU)** is handled by **Uniswap V2** on Ethereum, using the USDC/CRU pool (token `0x32a7C02e79c4ea1008dD6564b35F131428673c41`).\n+- **Final step (Ethereum CRU → Crust CRU)** is handled by **Crust’s Ethereum bridge**; mnemospark’s Crust wallet then uses native CRU to place storage orders / pins.\n+\n+Users never see CRU or any cross-chain UX. They just see: “pay USDC on Base → get persistent storage (IPFS+Crust).”\n+\n+---\n+\n+## 2. End-to-end flow (Option C with Superbridge)\n+\n+### 2.1 User-facing flow (Base + x402)\n+\n+1. **Agent/user requests storage** via mnemospark (e.g. upload file / sync state).\n+2. **Gateway returns 402** with a quote in **USDC on Base** (activity fee + storage fee, denominated in USDC).\n+3. **User/agent pays x402** from their Base wallet (USDC). Payment settles on Base.\n+4. From the user’s perspective, storage is now “paid for”; they do not see any bridging or CRU.\n+\n+### 2.2 Backend/relayer flow (USDC Base → CRU Crust)\n+\n+The backend (or a dedicated relayer) periodically or on-demand converts Base USDC into Crust CRU using three legs:\n+\n+1. **Base USDC → Ethereum USDC (Superbridge + CCTP)**\n+ - Use **Superbridge** to perform an L2→L1 **withdrawal** from Base to Ethereum, specifically for **USDC**, leveraging **Circle’s CCTP** for native USDC issuance. \n+ - Origin: Base. Destination: Ethereum mainnet. Token: USDC. \n+ - CCTP semantics: **burn USDC on Base**, **mint USDC on Ethereum**. \n+ - Timing: withdrawals from rollups to Ethereum typically have a **fraud-proof delay (~7 days)**; Superbridge can also offer **“Easy Mode”** (they relay/finalize for a fee) as per their docs.\n+\n+2. **Ethereum USDC → Ethereum CRU (Uniswap V2)**\n+ - On Ethereum, backend holds native USDC after the bridge finalizes. \n+ - It interacts with the **Uniswap V2 Router02** contract (e.g. `swapExactTokensForTokens`) using the USDC/CRU pool. \n+ - Path: `[USDC, CRU]` (or via WETH if needed). \n+ - It sets a conservative `amountOutMin` based on a quote (Uniswap SDK or aggregator) to control slippage and reduce sandwich risk.\n+\n+3. **Ethereum CRU → Crust CRU (Crust bridge)**\n+ - From Ethereum, the backend uses **Crust’s Ethereum bridge** to move CRU to the **Crust chain**. \n+ - Result: mnemospark’s **Crust wallet** holds native CRU, which is then used to place storage orders (pins) for user CIDs.\n+\n+This entire pipeline is **backend-only**; the user’s experience remains “I paid USDC on Base.”\n+\n+---\n+\n+## 3. Components & responsibilities\n+\n+### 3.1 x402 gateway (Base)\n+\n+- Handles **402 payment-as-authentication** on **Base**. \n+- Quotes storage operations in **USDC**. \n+- On successful x402 payment, records that the user has paid for a given storage operation or quota.\n+\n+### 3.2 Superbridge (Base ↔ Ethereum)\n+\n+- Used **only by mnemospark’s backend/relayer**, not directly by users. \n+- Bridges **USDC from Base to Ethereum** using **native USDC (CCTP)**: burn on Base, mint on Ethereum ([Native USDC docs](https://docs.superbridge.app/native-usdc)). \n+- Can be driven via their documented flows and, if needed, enterprise/whitelabel APIs.\n+\n+### 3.3 Uniswap V2 (Ethereum swaps)\n+\n+- Used by backend to swap **Ethereum USDC → CRU** on Uniswap V2’s Router02. \n+- Needs:\n+ - USDC balance on Ethereum.\n+ - Router02 approvals. \n+ - Safe `amountOutMin` and slippage configuration.\n+\n+### 3.4 Crust bridge + IPFS pinning\n+\n+- Backend uses Crust’s Ethereum bridge to move CRU to the Crust chain. \n+- Storage itself is handled by **IPFS W3Auth Gateway + Pinning Service** (per Crust docs):\n+ - Upload file via IPFS gateway → get **CID**. \n+ - Call IPFS W3Auth pinning service (with Web3Auth auth header) to **place storage order** on Crust. \n+ - Crust nodes replicate and persist the content.\n+\n+mnemospark’s Crust wallet funds these storage orders with the CRU obtained via the Option C pipeline.\n+\n+---\n+\n+## 4. Batching and treasury strategy\n+\n+- The pipeline does **not** need to run per-request. Instead:\n+ - The backend can **accumulate USDC on Base** from many x402 payments. \n+ - Periodically (or when a threshold is hit) it initiates a **Superbridge withdrawal** (Base→Ethereum USDC). \n+ - After the bridge finalizes, it performs a batch **USDC→CRU swap** on Uniswap, then bridges CRU to Crust.\n+- This reduces:\n+ - Bridge fees and gas costs. \n+ - Exposure to small-swap slippage. \n+ - Operational complexity (fewer moving pieces per user action).\n+\n+mnemospark effectively runs a **CRU treasury** on Crust, refilled via this automated on-chain route.\n+\n+---\n+\n+## 5. Pros and cons vs AWS-based MVP\n+\n+### Pros\n+\n+- **No AWS accounts or Orgs**: No management account, no sub-accounts, no CloudFormation stacks for each tenant. \n+- **Single user currency:** Users only ever pay in **USDC on Base**; aligns cleanly with x402 and Base ecosystem. \n+- **On-chain and composable:** Bridging and swapping are on-chain (Superbridge + Uniswap + Crust bridge); auditable and automatable. \n+- **Decentralized storage:** IPFS + Crust provides decentralized persistence rather than centralized S3 buckets.\n+\n+### Cons / added complexity\n+\n+- **Bridge latency:** L2→L1 (Base→Ethereum) via Superbridge/CCTP can involve **fraud-proof delays** for standard withdrawals; “Easy Mode” reduces UX friction but adds cost and dependency. \n+- **Liquidity & price risk:** USDC→CRU swaps depend on Uniswap pool liquidity and market price; need pricing + slippage strategy. \n+- **Two bridges:** Superbridge (Base↔Ethereum) + Crust bridge (Ethereum↔Crust) need monitoring and operational playbooks. \n+- **Treasury management:** mnemospark must manage CRU balances on Crust (and possibly on Ethereum), though this can be batched.\n+\n+Net: Option C **removes AWS infra/account complexity** but **adds a structured USDC→CRU on-chain pipeline**. For a Base-centric, “agents pay with USDC only” story, this is often a better product fit despite the new on-chain plumbing.\n+\n+---\n+\n+## 6. Open questions for future spec updates\n+\n+1. **Bridge timing guarantees:** Which Superbridge modes are acceptable (standard vs Easy Mode)? How do we reflect bridge delays in pricing and treasury planning? \n+2. **Minimum batching thresholds:** How much USDC should we accumulate on Base before triggering a bridge + swap? \n+3. **CRU price volatility:** How do we buffer USDC prices (markup, margin) to stay solvent if CRU/USDC moves between batch conversions? \n+4. **Failure handling:** What’s the playbook if any step fails (Superbridge withdrawal delayed, Uniswap pool illiquid, Crust bridge issues)? \n+5. **User messaging:** Do we surface any notion of “on-chain persistence via Crust/IPFS” to users, or keep it completely abstracted as “cloud storage paid in USDC”? \n+\n+This document is a reference for Option C and can be used to update the main product spec or create a dedicated “IPFS + Crust storage MVP” variant when you’re ready.\n+\n*** End Patch"}} ***!
