# MVP Option: AWS S3 + Client-Held Encryption (1 Account, 1 Bucket per Wallet)

**Status:** MVP option for product consideration  
**Context:** Single AWS account, one S3 bucket per customer wallet, encrypt data at rest without mnemospark managing keys. Customer’s key lives in their OpenClaw instance or in a secrets manager (e.g. 1Password).  
**Related:** [feature-02-s3-storage-backend.md](features/feature-02-s3-storage-backend.md), [mnemospark_product_spec_v2.md](mnemospark_product_spec_v2.md)

---

## 1. Architecture at a glance

- **1 AWS account** — No Organizations, no sub-accounts. Single account for all customers.
- **1 S3 bucket per customer wallet** — Bucket naming/keyed by wallet id (e.g. `mnemospark-<wallet-id>` or hashed). Clear isolation per wallet.
- **Encryption:** Data stored in S3 is **always ciphertext**. Mnemospark **never persists** the customer’s key. **Their data, their key.**
- **Key storage:** The decryption key is returned once to the **OpenClaw instance** and stored there—either in a **file** under `~/.openclaw/mnemospark/keys/` or in a **secrets manager** (e.g. 1Password, Bitwarden) via a configurable key-store backend.

---

## 2. Encryption model: envelope encryption

### 2.1 Roles of the keys

- **KEK (Key Encryption Key):** One per wallet. Used to wrap/unwrap per-object keys. **Generated once**, sent to the client in the API response, **never stored by mnemospark**. Client stores it (file or 1Password/similar).
- **DEK (Data Encryption Key):** One per object. Used to encrypt that object’s payload. Generated by the **client** per upload, used to encrypt the body, then **wrapped** with the KEK. Mnemospark stores only **ciphertext + wrapped_DEK** (in S3 object body or metadata).

### 2.2 Who does what

| Step               | Mnemospark (backend)                                       | Client (OpenClaw plugin/agent)                                                           |
| ------------------ | ---------------------------------------------------------- | ---------------------------------------------------------------------------------------- |
| **First-time key** | Generate KEK, return in response, do not persist           | Receive KEK, store in configured key store (file or 1Password)                           |
| **Upload**         | Receive ciphertext + wrapped_DEK; verify x402; write to S3 | Generate DEK; encrypt payload with DEK; wrap DEK with KEK; send ciphertext + wrapped_DEK |
| **Download**       | Read from S3; return ciphertext + wrapped_DEK              | Unwrap DEK with KEK; decrypt ciphertext                                                  |
| **Key storage**    | Never stores KEK                                           | Holds KEK in file or secrets manager (e.g. 1Password)                                    |

Mnemospark never sees plaintext or the KEK after the first response. Client never sends the KEK again; they only send ciphertext and wrapped DEKs.

## 3. Flows

### 3.1 First-time key provisioning

1. Client calls a “provision storage” or first storage endpoint (after x402 payment if required).
2. Backend checks whether this wallet already has a key on file (optional server-side “key id” or “provisioned” flag only—no key material). If not:
   - Backend generates a **KEK** (e.g. 256-bit).
   - Returns KEK in the response body (over TLS; same authenticated channel as x402).
   - Does **not** persist the KEK in S3, DB, or config.
3. Client receives KEK and **stores** it using the configured key store:
   - **File backend:** Write to e.g. `~/.openclaw/mnemospark/keys/<wallet_id>.key` (restrict permissions, e.g. `chmod 600`).
   - **1Password (or similar):** Write to a Secure Note or custom item via 1Password CLI or Connect API (see Section 5).
4. For any subsequent request, client loads KEK from the key store when needed; backend never sees KEK again.

### 3.2 Upload (add more data)

1. Client loads KEK from key store (file or 1Password).
2. Client generates a new **DEK** for this object.
3. Client encrypts the request body with DEK → **ciphertext**.
4. Client wraps DEK with KEK → **wrapped_DEK**.
5. Client sends to backend: **ciphertext** + **wrapped_DEK** (and object key/path). Optionally includes idempotency key.
6. Backend verifies x402 (if required), then writes to the wallet’s S3 bucket:
   - Object content = ciphertext (with wrapped_DEK in object metadata, e.g. `x-amz-meta-wrapped-dek`, or prepended to body with fixed layout).
7. Backend never sees KEK or plaintext.

### 3.3 Download

1. Client requests object (with x402 if required).
2. Backend reads from S3, returns **ciphertext** + **wrapped_DEK** (from body/metadata).
3. Client loads KEK from key store.
4. Client unwraps DEK with KEK, then decrypts ciphertext with DEK → plaintext.
5. Client returns plaintext to the agent/user.

---

## 4. S3 object layout

- **Option A (recommended):** Object body = **ciphertext** only. Store **wrapped_DEK** in S3 object **user metadata** (e.g. `x-amz-meta-wrapped-dek` as base64). Client and gateway agree on this; client reads metadata + body and unwraps/decrypts.
- **Option B:** Object body = **fixed-length wrapped_DEK** (e.g. 40 bytes) concatenated with **ciphertext**. Client splits and decrypts. No metadata needed.

Either way, mnemospark never stores the KEK or plaintext.

---

## 5. Key store: file vs 1Password (or similar)

The plugin supports a **key store** abstraction so the KEK can live in a file or in an external secrets manager.

### 5.1 File backend (default)

- **Path:** e.g. `~/.openclaw/mnemospark/keys/<wallet_id>.key` (or hashed wallet id).
- **Format:** Raw key bytes or base64 in a small file; permissions restricted to owner (e.g. `chmod 600`).
- **Backup:** Document that the user should back up this key; if lost, data in S3 is unrecoverable.

### 5.2 1Password (or similar) backend

- **Storage:** KEK is stored in 1Password (e.g. Secure Note or custom item) under a vault/item the user or plugin configures.
- **Write (first time):** When backend returns the KEK, plugin can call 1Password CLI (e.g. `op item create`) or 1Password Connect API to create/update the item with the key. Alternatively, plugin can show the key once and the user pastes it into 1Password manually.
- **Read (every upload/download):** Plugin loads KEK by calling e.g. `op read "op://VaultName/ItemName/field"` or 1Password Connect API. Requires:
  - 1Password CLI (`op`) in PATH, or Connect server + token.
  - Config: vault name, item name (or reference), and for Connect: server URL + API token.
- **Same pattern** works for Bitwarden (CLI/API), HashiCorp Vault, macOS Keychain, Windows Credential Manager, or `pass`, by adding more key-store backends.

### 5.3 Config example

```json
{
  "mnemospark": {
    "storage": {
      "keyStore": "file",
      "keyStorePath": "~/.openclaw/mnemospark/keys"
    }
  }
}
```

Or for 1Password:

```json
{
  "mnemospark": {
    "storage": {
      "keyStore": "1password",
      "1password": {
        "vault": "Private",
        "item": "mnemospark-wallet-xyz",
        "field": "key"
      }
    }
  }
}
```

(Exact keys and Connect vs CLI can be refined in implementation.)

## 6. Edge cases and operational notes

| Scenario                                      | Handling                                                                                                                                                                                                           |
| --------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Client loses KEK**                          | Data in S3 is unrecoverable. Document clearly; optionally support “rotate storage” (new KEK, re-encrypt) if user still has key elsewhere.                                                                          |
| **Key compromise**                            | Support “rotate KEK”: generate new KEK, return to client; client re-wraps DEKs for existing objects (or re-uploads); old KEK retired.                                                                              |
| **Multiple OpenClaw instances (same wallet)** | They must share the same KEK (e.g. same key file or same 1Password item). User responsibility to sync/back up.                                                                                                     |
| **First request and no key yet**              | Backend generates KEK, returns in response; client stores and uses for that and future requests. If client already has a key (e.g. from another device), backend can skip generation and client uses existing KEK. |
| **S3 bucket doesn’t exist**                   | Create bucket per wallet on first write (or via provisioning step); naming e.g. `mnemospark-<account-id>-<wallet-id-hash>`.                                                                                        |

---

## 7. Success criteria for this option

- One AWS account; one S3 bucket per wallet; no Organizations or sub-accounts.
- All object data in S3 is ciphertext; object metadata (or body layout) includes wrapped_DEK only.
- KEK is generated once by backend, returned in API response, never persisted by mnemospark.
- Client stores KEK in file or 1Password (or similar) via configurable key store.
- Upload: client sends ciphertext + wrapped_DEK; download: backend returns ciphertext + wrapped_DEK; client decrypts with KEK.
- Document backup and loss-of-key implications for users.
